Understanding CI/CD in the AQC Project

A narrative guide to how code becomes running software — and how you can run it yourself.

This guide walks you through the project’s CI/CD and Docker setup as a single story: what runs when, why it’s structured this way, and how to bring the same environments up on your machine with Docker Desktop or the command line. No list-dumping — just explanations you can follow from start to finish.

1. The big picture: what CI/CD is doing

In this repo, Continuous Integration (CI) means: every time someone pushes code or opens a pull request, automated jobs run to install dependencies, check types, lint, test, and build the backend and frontend. That way the team finds breakage quickly instead of discovering it later in production.

Continuous Deployment (CD) is only partly set up. The backend workflow is prepared to build a Docker image on the main branch and tag it for GitHub’s container registry (GHCR), but nobody has wired in registry login and docker push yet. So today the pipeline stops at “build and optionally tag”; actually pushing and deploying is something you’d add when you’re ready.

The project has two main applications: a backend (Node.js, Express, TypeScript, MongoDB) under backend/, and a frontend (React, Vite) under Lunara/. The pipelines are split so that changing only backend code doesn’t run frontend tests, and vice versa. That saves time and keeps feedback focused.

2. When does what run?

GitHub Actions decides whether to run a workflow based on the event (push, pull_request, etc.) and, for our workflows, on which files changed. You can think of it like this: a workflow runs only if the set of changed files overlaps with the paths it cares about.

In set terms: if C is the set of files changed in the push or PR, and P is the set of paths the workflow watches, then the workflow runs when C ∩ P ≠ ∅ (that is, at least one file is in both sets). For the backend workflow, P is { backend/**, .github/workflows/backend-ci.yml }. So editing backend/src/server.ts puts that file in C, and since it matches backend/**, the backend CI runs. Editing only Lunara/src/App.tsx does not match P for the backend, so backend CI does not run.

The frontend workflow uses the same idea, but with P set to { Lunara/**, .github/workflows/frontend-ci.yml }. The SonarQube workflow (build.yml) doesn’t use path filters; it runs on every push and pull request that targets main.

Important

All three workflows are currently turned off with a condition if: false in the YAML (often to save GitHub Actions minutes). To use them, open each workflow file and remove that line so the job actually runs.

3. The backend pipeline, step by step

File: .github/workflows/backend-ci.yml. When this workflow runs, GitHub spins up a fresh Ubuntu runner and runs one job, but that job is matrixed: it runs twice, once with Node 18 and once with Node 20. So you get two parallel pipelines, both doing the same steps in the backend/ directory. That way you know the backend works on both LTS versions.

The sequence is designed to fail fast. First the repo is checked out. Then Node is installed with npm’s cache pointed at backend/package-lock.json, so repeated runs reuse the cache when the lockfile hasn’t changed. Next, npm ci installs dependencies exactly as in the lockfile — no floating versions. Then:

Type-check. npm run type-check runs TypeScript’s compiler in “no emit” mode (tsc --noEmit). It doesn’t write files; it only checks that types are consistent. If something is mis-typed, the step fails and the rest of the job can be skipped.
Tests. npm test -- --coverage runs Jest and produces a coverage report under backend/coverage. That folder is then uploaded as a workflow artifact (e.g. backend-coverage-18.x, backend-coverage-20.x) so you can download it from the Actions run and inspect or send it to a service like Codecov later.
Docker build (only on main). If the branch is main, the workflow builds a Docker image from backend/. The tag is ghcr.io/<owner/repo>:<git-sha>, with the repo name lowercased because Docker image names must be lowercase. No docker push is done unless you add a step that logs into GHCR and pushes — so the image exists only on the runner and is discarded when the job ends. Adding that push step is how you’d complete the “CD” part for the backend.

So in one sentence: backend CI installs, type-checks, tests with coverage, saves the coverage artifact, and on main builds a Docker image tagged by commit SHA.

4. The frontend pipeline, step by step

File: .github/workflows/frontend-ci.yml. The structure mirrors the backend: path filter on Lunara/** (and the workflow file itself), matrix on Node 18 and 20, default working directory Lunara.

After checkout and Node setup (with npm cache for Lunara/package-lock.json), the job runs npm ci, then npm run lint (ESLint). If lint fails, the job fails — no point building broken style or patterns. Then npm test -- --coverage runs the frontend test suite and produces coverage under Lunara/coverage. Next, npm run build runs the Vite + TypeScript build and writes the production bundle to Lunara/dist. That directory is the “compiled” frontend: static HTML, JS, and CSS that you could serve with any web server or with the serve package as in the frontend Dockerfile.

The workflow then uploads two artifacts: Lunara/dist as frontend-dist-<node-version> and Lunara/coverage as frontend-coverage-<node-version>. The dist artifact is what a deployment step would use if you later add one (e.g. upload to S3 or deploy to a host). There is no Docker build in the frontend workflow today; the frontend image is only built when you run Docker locally or from another process.

5. SonarQube and the build workflow

File: .github/workflows/build.yml. This workflow runs on every push and pull request to main, with no path filter. It checks out the repo with full history (for Sonar’s analysis), installs Node 18, and installs the Sonar scanner (@sonar/scan). Then it runs SonarQube analysis twice: once from backend/ with project key AQC-Backend, and once from Lunara with project key AQC-Frontend.

Both Sonar steps use continue-on-error: true and only run the scanner when the secrets SONAR_HOST_URL and SONAR_TOKEN are set. If you haven’t configured a SonarQube server and added those secrets in the repo, the steps effectively no-op (they may print “SonarQube secrets not configured, skipping analysis”) and the workflow still succeeds. So the workflow is optional: it becomes useful when you point it at a real SonarQube instance for code quality and security reports.

6. How the app gets packaged: Docker

Docker turns your application and its runtime into a single artifact: an image. Anyone with that image can run the same app without installing Node or matching your machine. The repo has two Dockerfiles — one for the backend, one for the frontend — and both use multi-stage builds so the final image doesn’t contain build tools or dev dependencies.

Backend image (backend/Dockerfile)

The backend is TypeScript, so it must be compiled to JavaScript before Node can run it. The Dockerfile does that in a build stage named build: it uses node:18-alpine, copies package*.json, tsconfig.json, and src/, runs npm ci and npm run build, and ends up with compiled output in dist/. That stage is temporary.

The runtime stage starts from a fresh node:18-alpine. It copies package*.json, runs npm ci --only=production (no devDependencies), and copies the compiled output from the build stage. One practical detail: the runtime image also includes the src/ folder so Swagger can read the @swagger annotations (the API is compiled to dist/, but the docs are generated by scanning src/**/*.ts). The Dockerfile then creates a non-root user lunara, switches to it, exposes port 10000, and declares a health check that hits GET /api/health every 30 seconds. The process is started with CMD ["npm", "start"], which runs node dist/server.js. So the backend in Docker listens on port 10000 and expects environment variables like MONGODB_URI, JWT_SECRET, and the rest that server.ts reads.

Frontend image (Lunara/Dockerfile)

The frontend is a Vite app: you run npm run build and get a dist/ folder of static files. The Dockerfile’s build stage uses node:20-alpine, copies the app, and runs the build. Because Vite bakes environment variables into the build output, the Dockerfile supports passing VITE_API_BASE_URL at build time so the built frontend points at the right API. The prod stage copies dist/ into a new image, installs the serve package globally, and runs serve -s dist -l 5173. So the container serves the built frontend on port 5173.

7. Running multiple services: Docker Compose

Docker Compose is for running several containers together with a single command: the app services (backend + frontend), the app’s database (MongoDB), and optional tooling (SonarQube backed by Postgres). The repo’s docker-compose.yml defines a shared network lunara-net and the services you need for a “pull a PR, run it, test it” workflow.

The db service uses the official postgres:17 image. In this project it exists to back SonarQube, not the application. It exposes 5432 and stores its data in db_data so it survives restarts.

The mongo service is the application database. The backend is a Node/Express app that expects MONGODB_URI, and in Compose we point it at mongodb://mongo:27017/lunara.

The backend service is built from ./backend and exposed on port 10000. A few small “quality of life” values are set automatically in Compose so Docker feels like local dev:

  • Swagger server dropdown: API_URL is set to http://localhost:10000/api, so /api-docs points at the Docker backend by default.
  • CORS: the backend allowlist includes http://localhost:10000 so “Try it out” works from Swagger when running in Docker.
  • Email placeholders: EMAIL_USER/EMAIL_PASS are non-empty in Compose so the backend can boot even if you aren’t sending email during local testing.

The frontend service builds ./Lunara and serves it on port 5173. The build passes VITE_API_BASE_URL so the frontend talks to the Docker backend by default.

The sonarqube service runs the community SonarQube image, uses the Postgres db for its database, and exposes 9000. Its data and extensions live in named volumes so they persist.

8. You run it: spinning up environments

You need Docker Desktop (or Docker Engine plus Compose) and the repo cloned. All commands below assume you’re in the repo root (e.g. e:\AQC).

Run full app (backend + frontend)

To run only the app services and skip Postgres/SonarQube, set MONGODB_URI (in your shell or in a .env file in the repo root) and start backend and frontend:

docker compose up -d backend frontend

The backend uses MONGODB_URI from your environment (e.g. Atlas or mongodb://mongo:27017/lunara if you also start mongo). To bring up the entire stack — mongo, backend, frontend, Postgres, and SonarQube — use:

docker compose up -d

After code changes, rebuild the images and start again:

docker compose build backend frontend
docker compose up -d backend frontend

URLs when running: Frontend http://localhost:5173 · Backend health http://localhost:10000/api/health · Swagger http://localhost:10000/api-docs · SonarQube http://localhost:9000 (if you started the full stack).

The standard “pull a PR, run it in containers” loop

This is the workflow you’ll use again and again as you pull your partner’s PRs:

git checkout <pr-branch>
docker compose build backend frontend
docker compose up -d

At this point you have a full local environment running your checked-out branch:

  • Frontend: http://localhost:5173
  • Backend health: http://localhost:10000/api/health
  • Swagger UI: http://localhost:10000/api-docs
  • SonarQube: http://localhost:9000

If you want to watch what a service is doing, use Compose logs:

docker compose ps
docker compose logs -f backend
docker compose logs -f frontend

Building and running images by hand

From the repo root you can build the backend and frontend images without Compose:

docker build -t lunara-backend:local -f backend/Dockerfile backend
docker build -t lunara-frontend:local -f Lunara/Dockerfile Lunara

To run the backend alone (e.g. with MongoDB already running on your host), use:

docker run --rm -p 10000:10000 ^
  -e MONGODB_URI=mongodb://host.docker.internal:27017/lunara ^
  -e JWT_SECRET=your-secret ^
  -e JWT_REFRESH_SECRET=your-refresh-secret ^
  -e FRONTEND_URL=http://localhost:5173 ^
  -e EMAIL_USER= ^
  -e EMAIL_PASS= ^
  lunara-backend:local

(On PowerShell you may need to use backticks for line continuation instead of ^.) On Linux, if host.docker.internal isn’t available, use your host’s IP or run MongoDB in a container on the same Docker network.

9. Configuration and secrets

The backend (backend/src/server.ts) requires these environment variables: JWT_SECRET, JWT_REFRESH_SECRET, MONGODB_URI, EMAIL_USER, EMAIL_PASS, FRONTEND_URL. Optional ones include PORT (default 10000), API_URL (used by Swagger’s server dropdown), CORS_ALLOWED_ORIGINS (to override the allowlist), and EMAIL_PORT. The frontend uses VITE_API_BASE_URL to point at the API; in Docker this is passed as a build argument so the built frontend targets the containerized backend.

Controlled account creation (admin bootstrap)

If you don’t want public account creation, you still need a way to create the first admin user. This repo includes a small bootstrap script:

cd backend
npm run create:admin

Once you have an admin user, log in via POST /api/auth/login to get a JWT, then create provider accounts via the admin-only endpoint POST /api/admin/providers.

For GitHub Actions: to enable SonarQube in build.yml, add repository secrets SONAR_HOST_URL and SONAR_TOKEN. To push the backend image to GHCR from backend-ci.yml, add a step that logs in (e.g. with GITHUB_TOKEN or a PAT with write:packages) and runs docker push after the build step.

For Docker Compose, use docker-compose.yml.example as a reference and replace placeholders like <YOUR_DB_USER>, <YOUR_DB_PASSWORD>, <YOUR_JWT_SECRET>. You can put overrides in docker-compose.override.yml or in a .env file in the repo root so you don’t edit the main compose file.
