## Integrity Position Statement

All claims made in this capstone paper packet are grounded in verifiable repository evidence and documented project artifacts. The project team made a deliberate effort throughout the development lifecycle to avoid overstating scope, status, or individual contribution, and to ensure that every technical assertion can be traced back to implemented code, test results, or project documentation.

## Evidence-Based Reporting

Technical claims throughout these papers are anchored to concrete implementation artifacts rather than aspirational descriptions. When this packet states that LUNARA implements nineteen backend route modules, that figure is derived from the actual route files in `backend/src/routes`. When it reports 1044 Jest tests with backend and frontend statement coverage figures (90.58% and 63.35%, respectively, in Milestone 5), those numbers come from test runner output and coverage reports in `05_milestone_5.tex`. SonarQube aggregate coverage (e.g., 81.9%) is reported separately from Jest/Istanbul. When it describes SonarQube A/A/A ratings, those ratings are taken from the quality analysis workflow documented in `Docs/SCA_GUIDE.md`. This approach ensures that the capstone papers serve as an honest reflection of what was built rather than a marketing narrative.

## Contribution Transparency

The project was developed collaboratively, and team roles are represented consistently with the documented project context. Where specific contributions are attributable to individual team members, they are described accurately. Where work was shared or iterative, it is presented as a collaborative effort. The Git commit history provides an auditable record of who contributed what and when, supporting any reviewer who wishes to verify the contribution narrative.

## Scope Honesty

The project is described throughout this packet as being in the maintenance stage, which accurately reflects the current lifecycle position. Features that were implemented and tested are presented as completed work. Features that were formally deferred—digital journaling, daily insights, sleep and feeding trackers, and AI note summarization—are explicitly identified as post-launch roadmap items with documented justification for their deferral. At no point does this packet represent planned or future capabilities as delivered functionality.

## Reproducibility and Auditability

The source control history on GitHub provides a verifiable timeline of implementation progression, from initial scaffolding through feature construction, testing, and release preparation. The automated test suite can be executed by any reviewer with access to the repository to confirm the reported test counts and pass rates. The documentation set—README files, development guide, sprint records, and API documentation—provides the context necessary for an auditor to understand the project's scope and verify its claims without requiring oral explanation.

## Integrity in Maintenance

Academic integrity does not end at submission. As the project continues in its maintenance stage, the same principles apply: release notes will accurately describe changes, bug reports and patch records will reflect actual defects and their resolutions, and any future enhancement will be clearly distinguished from the functionality that was delivered during the capstone period.

For detailed information regarding the University's academic integrity policies, refer to the GCU Academic Integrity webpage (Grand Canyon University, 2015).
